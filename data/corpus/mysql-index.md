# MySQL 索引原理

索引是数据库最重要的性能武器。没有索引，一次查询要逐行扫描整张表（全表扫描）；
有了合适的索引，查询能像查字典一样快速定位。MySQL（InnoDB 引擎）的索引底层是 B+ 树。

## 为什么是 B+ 树

B+ 树是一棵多路平衡搜索树，专为磁盘存储设计：

- **矮胖**：一个节点就是一个数据页（默认 16KB），能存很多键，所以树很矮，
  通常三到四层就能索引千万级数据，查一次只需三四次磁盘 IO。
- **只有叶子节点存数据**，非叶子节点只存索引键用于导航，进一步提升每层的分叉数。
- **叶子节点用双向链表串联**，天然支持范围查询（`between`、`>`、`order by`），
  这是哈希索引做不到的——哈希只能等值查询。

## 聚簇索引与二级索引

InnoDB 的主键索引是**聚簇索引**：叶子节点直接存放整行数据，数据和索引存在一起。
除主键外建立的索引叫**二级索引**（非聚簇索引），它的叶子节点存的不是整行，而是主键值。

## 回表与覆盖索引

用二级索引查询时，先在二级索引树上找到主键值，再拿主键值回到聚簇索引里查完整行数据——
这个"二次查找"的过程叫**回表**。回表要多走一棵树，是额外开销。

如果一个查询需要的字段，恰好都在使用的索引里（比如 `select id, name` 且 name 上有索引），
就不需要回表，直接从索引拿到全部数据返回，这叫**覆盖索引**。
"能覆盖就不回表"是索引优化的核心思路之一，面试常问怎么优化慢查询，覆盖索引是标准答案。

## 联合索引与最左匹配

在多个字段上建的索引叫**联合索引**（如 `(a, b, c)`）。它遵循**最左匹配原则**：
必须从最左列开始、连续地使用，索引才生效。`where a=1 and b=2` 能用，`where b=2` 用不上，
`where a=1 and c=3` 只有 a 部分生效。原因是联合索引按 a、再 b、再 c 的顺序排序，
跳过 a 直接找 b，数据在树里是无序的，只能全表扫描。

## 索引为什么没生效

一个高频排查场景：给字段加了索引，查询还是很慢、走了全表扫描。常见原因：

- 违反最左匹配，或对索引列做了函数/运算（`where year(t)=2024`）导致索引失效。
- 隐式类型转换（字段是字符串却用数字查）。
- 用 `like '%xx'` 前导通配，或区分度太低（如性别字段）优化器判断走索引还不如全表快。

**索引下推**（ICP）是 MySQL 5.6 的优化：把 where 条件的过滤下推到存储引擎层，
在遍历索引时就先过滤掉不满足的行，减少回表次数。

## Buffer Pool 缓冲池

磁盘 IO 是数据库最慢的环节，InnoDB 用 **Buffer Pool（缓冲池）** 把热点数据页缓存在内存里，
读写先走内存，命中缓冲池就不必碰磁盘。缓冲池命中率越高，查询延迟越低、吞吐越高，
这是数据库层面最重要的缓存。它用 LRU 链表管理哪些数据页留在内存、哪些被淘汰，
和浏览器缓存"把常用资源留在近处、减少远端访问"是同一种朴素思想，只是发生在数据库内部。
