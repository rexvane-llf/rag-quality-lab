# HTTP 缓存机制

HTTP 缓存是 Web 性能优化的第一道防线。核心目标：让浏览器在第二次访问同一个网页时，
尽量不重新下载资源，直接复用本地副本，从而几乎瞬间加载。缓存分为强缓存与协商缓存两层，
配合 CDN 边缘节点，可以把绝大多数请求挡在离用户最近的地方。

## 强缓存

强缓存命中时，浏览器直接从本地磁盘或内存读取资源，根本不向服务器发起请求，
在开发者工具里表现为 `200 (from disk cache)` 或 `from memory cache`。这是最快的一档。

控制强缓存的两个响应头：

- `Expires`：HTTP/1.0 的产物，值是一个绝对过期时间点（GMT 时间）。缺点是依赖客户端本地时钟，
  用户改了系统时间就会失效，已基本被淘汰。
- `Cache-Control`：HTTP/1.1 的主力，用相对时间，优先级高于 `Expires`。常见指令：
  - `max-age=31536000`：资源在 31536000 秒（一年）内都算新鲜，直接用本地副本。
  - `no-cache`：每次都要走协商缓存问一下服务器，**不是不缓存**（这是高频面试陷阱）。
  - `no-store`：真正的不缓存，任何副本都不留，用于敏感数据。
  - `private` / `public`：能否被 CDN 等中间代理缓存。

## 协商缓存

当强缓存过期后，浏览器不会立刻重新下载，而是带着"校验凭证"去问服务器：我手里这份还能用吗？
如果资源没变，服务器回一个 `304 Not Modified`（响应体为空），浏览器继续用本地副本；
只有真的变了才返回 `200` 和完整内容。这样即使过期，也能省下传输大文件的带宽。

两组校验字段：

- `Last-Modified` / `If-Modified-Since`：基于最后修改时间。浏览器把上次拿到的 `Last-Modified`
  通过 `If-Modified-Since` 带回去比对。缺点是精度只到秒，且"内容没变但时间戳变了"会误判。
- `ETag` / `If-None-Match`：基于内容指纹（哈希）。浏览器把 `ETag` 通过 `If-None-Match` 带回，
  服务器比对指纹。精度更高，优先级高于 `Last-Modified`，是解决秒级精度问题的方案。

## CDN 与缓存层级

CDN（内容分发网络）在地理上把静态资源缓存到离用户最近的边缘节点。用户请求先打到边缘节点，
命中则直接返回，未命中才**回源**到源站拉取再缓存下来。这一层缓存承担了海量图片、JS、CSS 的分发，
极大降低了源站压力与用户的网络延迟。

缓存更新的经典难题：文件更新了但用户还拿着旧缓存。工程上的标准解法是**内容哈希文件名**
（如 `app.8f3a2c.js`），内容一变文件名就变，配合 `max-age` 一年的强缓存，既长期缓存又能即时更新——
这就是"缓存+ 指纹"组合拳。
